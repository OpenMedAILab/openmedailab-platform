# 认领审批中与撤回入口设计方案

更新时间：2026-06-18

状态：已执行，待最终产品验收

关联文档：`docs/superpowers/specs/2026-06-18-project-card-self-relation-labels-design.md`

适用范围：课题展示页卡片操作区、课题详情页同款操作区、本人关系状态回显、无法认领原因提示。

## 背景

当前课题卡片操作区包含 `认领项目负责人` 和 `认领第一单位` 两个需要管理员审核的入口。领导反馈的核心问题是：用户提交认领后，如果按钮直接变成撤回入口，会让人误以为已经正式认领成功，无法区分“申请正在审批”和“审批已通过”。

现有产品基线已经明确：

- `认领项目负责人` 和 `认领课题（论文第一单位）` 提交后为 `pending`，需要管理员审批。
- 每个课题同一时间最多 1 个 active 项目负责人认领席位、1 个 active 论文第一单位认领席位；active 包含 `pending` 和 `approved`。
- 面向用户的撤回文案必须写明具体类型，不能只写 `撤回认领` 或 `撤回第一单位`。

本设计只补齐按钮状态、审批中提示和撤回入口展示规则，不改变席位唯一性、管理员审批、第一单位名称填写、积分检查和课题阶段限制。

## 产品判断

资深产品经理视角下，这个功能的关键不是多一个按钮文案，而是把“申请态”和“授权态”分离。

1. `pending` 是申请正在处理，不是用户已经获得项目负责人或论文第一单位资格。
2. `approved` 才是正式关系，只有这时才可以展示“撤回具体认领类型”的入口。
3. `pending` 虽然不是正式认领，但仍占用对应认领席位；其他用户尝试认领时必须看到“已有申请待审或已通过”的原因。
4. 如果未来需要支持用户撤销待审申请，文案应另行设计为 `撤回项目负责人申请` / `撤回论文第一单位申请`，不能与本次的通过后撤回认领混用。

因此本次推荐采用四类按钮状态：可提交、审批中、已通过可撤回、不可认领。

## 状态与文案

### 项目负责人认领

| 业务状态 | 按钮文案 | 按钮行为 | 反馈说明 |
| --- | --- | --- | --- |
| 未登录 | `认领项目负责人` | 点击触发登录引导或登录 toast | `登录后可提交项目负责人认领。` |
| 可提交 | `认领项目负责人` | 打开确认弹窗，确认后提交审核 | 成功 toast：`项目负责人认领已提交审核` |
| 提交中 | `提交中...` | 禁止重复提交 | 保持按钮宽度稳定 |
| 本人 `pending` | `项目负责人审批中` | 不提交、不撤回；hover/focus/移动端点击只展示说明 | `你的项目负责人认领正在管理员审批中，审批通过后可撤回项目负责人认领。` |
| 本人 `approved` | `撤回项目负责人认领` | 点击撤回该 approved 认领 | 成功 toast：`已撤回项目负责人认领` |
| 撤回中 | `撤回中...` | 禁止重复点击 | 失败 toast：`撤回项目负责人认领失败` |
| 他人 active 占用 | `暂不认领` | 不提交；展示原因提示 | `该课题已有项目负责人认领申请或已通过项目负责人认领。` |
| 阶段不可提交 | `暂不认领` | 不提交；展示原因提示 | `当前阶段暂不接受项目负责人认领。` |
| 积分不足 | `暂不认领` | 不提交；展示原因提示 | `当前积分不足，暂不能提交项目负责人认领。` |
| 数据冲突 | `暂不认领` | 不提交；展示原因提示 | `该课题的项目负责人认领数据存在冲突，请联系管理员处理。` |

## 交互设计

### 审批中按钮

审批中按钮是状态提示，不是主操作入口。

- 桌面端 hover 或键盘 focus 时，在按钮附近展示轻量 tooltip。
- 移动端点击审批中按钮时展示 toast 或底部小浮层，内容与桌面 tooltip 一致。
- 按钮不触发 `withdrawInteraction`，也不再次提交认领。
- 建议使用 `aria-disabled="true"` 保持语义，同时保留 focus 和提示能力；如果使用原生 `disabled`，必须额外提供可聚焦提示触发点。
- 视觉上使用低强调的待审样式，建议为中性色或浅金色描边，不使用绿色成功态，避免暗示已通过。

### 已通过撤回按钮

只有 `approved` 状态展示撤回入口。

- 项目负责人：`撤回项目负责人认领`
- 论文第一单位：`撤回论文第一单位认领`

按钮文案必须完整展示具体认领类型。窄屏时允许两行换行，不允许缩短为 `撤回认领` 或 `撤回第一单位`。

撤回行为延续现有轻量交互：点击后直接进入 `撤回中...`，成功后局部刷新该课题的按钮、本人关系、组队状态和状态卡，不刷新整页。

### 不可认领提示

不可认领不应只表现为灰按钮。用户需要知道为什么不能认领。

- 桌面端：hover/focus 到按钮时，tooltip 跟随按钮附近展示，距离触发按钮最近边缘不超过 12px。
- 移动端：点击不可认领按钮时展示 toast 或底部小浮层。
- 提示不渲染在可能被卡片 `overflow: hidden` 裁切的内部层级。
- Escape、失焦、鼠标移出或点击外部时关闭。

## API 与数据要求

当前 `claim_availability` 如果只返回 `action = "withdraw"`，前端无法区分本人 `pending` 和本人 `approved`。建议后端明确返回可用于渲染按钮的状态。

推荐 payload：

```json
{
  "leader": {
    "available": false,
    "action": "pending",
    "own_status": "pending",
    "own_interaction_id": 123,
    "reason_code": "own_pending",
    "reason": "你的项目负责人认领正在管理员审批中，审批通过后可撤回项目负责人认领。"
  },
  "paper_first_unit": {
    "available": false,
    "action": "withdraw",
    "own_status": "approved",
    "own_interaction_id": 456,
    "reason_code": "own_approved",
    "reason": "你的论文第一单位认领已通过，可撤回论文第一单位认领。"
  }
}
```

`action` 枚举建议：

| action | 含义 | 前端处理 |
| --- | --- | --- |
| `submit` | 当前用户可提交 | 展示提交按钮 |
| `pending` | 本人同类型认领待审批 | 展示审批中状态，不触发提交或撤回 |
| `withdraw` | 本人同类型认领已通过 | 展示通过后撤回按钮 |
| `unavailable` | 不可提交 | 展示 `暂不认领` 与原因提示 |

后端判定顺序建议：

1. 本人同类型 `pending`：返回 `action = "pending"`。
2. 本人同类型 `approved`：返回 `action = "withdraw"`。
3. 未登录：返回 `unavailable/login_required`。
4. 非开放招募或组队中：返回 `unavailable/stage_not_recruiting`。
5. 他人 active 占用或数据冲突：返回 `unavailable/slot_occupied` 或 `unavailable/data_conflict`。
6. 积分不足：返回 `unavailable/insufficient_credits`。
7. 其余可提交：返回 `submit/available`。

前端不要再用“active 关系”等价推导“可撤回”。需要拆成：

- `isClaimPending(request)`
- `isClaimApproved(request)`
- `claimCanWithdraw(request)`，仅 `approved` 返回 true。

## 与本人关系标签的关系

本设计主要约束操作区按钮，不强制修改上一版金色斜角标签和列表排序规则。

产品建议：

- `pending` 认领仍属于“与我有关”的课题，可继续用于优先排序和金色 hover 顶线，因为用户需要跟进审批结果。
- 但在任何需要表达最终关系的场景，不应把 `pending` 直接写成“已认领成功”。
- 如果后续要继续精细化卡片金色斜角标签，建议另开一版设计讨论 `我已认领` 是否只代表 `approved`。本次先通过按钮文案解决领导提出的主要困惑。

## 验收标准

### 功能验收

1. 本人项目负责人认领为 `pending` 时，课题卡按钮精确显示 `项目负责人审批中`。
2. 本人论文第一单位认领为 `pending` 时，课题卡按钮精确显示 `第一单位认领审批中`。
3. 两个 `pending` 按钮点击后不得调用撤回接口，也不得再次提交认领。
4. 本人项目负责人认领为 `approved` 时，按钮精确显示 `撤回项目负责人认领`。
5. 本人论文第一单位认领为 `approved` 时，按钮精确显示 `撤回论文第一单位认领`。
6. 撤回成功后，对应认领状态变为 `withdrawn`，按钮恢复为可提交或按当前不可认领原因展示，课题卡局部刷新。
7. 他人已有 `pending` 或 `approved` 同类型认领时，当前用户不能提交同类型认领，并看到对应原因提示。
8. `withdrawn` 和 `rejected` 不继续占用认领席位；如果其他条件允许，用户可重新提交。

### 文案验收

1. 前端面向用户文案不得出现泛称 `撤回认领`。
2. 前端面向用户文案不得出现泛称 `撤回第一单位`。
3. 审批中文案必须包含明确审核对象：项目负责人或第一单位。
4. 不可认领提示必须说明原因，不能只显示 `暂不认领`。

### 可访问性与响应式验收

1. 审批中和不可认领按钮可被键盘 focus，并能触发同等提示。
2. tooltip 不遮挡课题标题、主操作按钮和弹窗确认按钮。
3. 移动端无法 hover 时，点击审批中或不可认领按钮能看到同等说明。
4. 长文案按钮在 375px 宽度下不溢出、不截断关键类型词，必要时两行展示。
5. 按钮触达区域不小于 44px 高，按钮之间间距不小于 8px。

### 技术验收

1. `claim_availability` 能区分本人 `pending` 与本人 `approved`。
2. 前端不再把 `pending` 认领当作可撤回认领。
3. 前端单元测试覆盖 `pending -> 审批中`、`approved -> 撤回具体认领类型`、`unavailable -> 原因提示`。
4. 后端测试覆盖 `own_pending`、`own_approved`、`slot_occupied`、`stage_not_recruiting`、`insufficient_credits`。
5. 浏览器验收覆盖桌面 hover/focus、移动端点击提示、撤回成功后的局部刷新。

## 风险与开放问题

1. 待审批申请是否允许用户主动撤销：本设计默认首页卡片不提供 pending 撤销，避免和通过后撤回认领混淆。如业务必须允许撤销，建议放在个人中心或审批记录详情中，文案使用 `撤回申请`，不要使用 `撤回认领`。
2. 金色斜角标签是否区分 pending 和 approved：本设计暂不强制调整。若领导继续认为 `我已认领` 会误导，需要单独把 pending 标签设计为 `认领审批中` 或只在 approved 后展示 `我已认领`。
3. 审批时长预期：如果审批可能超过 24 小时，建议后续在个人中心展示申请时间和审核意见入口，减少用户重复点击卡片按钮。
